# AI Neovim Workbench Design

Date: 2026-04-29

## Purpose

Build a from-scratch Neovim configuration for an AI-heavy coding workflow. The goal is not a minimal editor and not a black-box distribution. The goal is a mature, stable, inspectable workbench that replaces the useful parts of VSCode while staying fast and controllable.

The primary workflow assumes most code is generated by AI tools. Neovim must make it easy to understand generated code, review diffs, inspect diagnostics, run tests, debug failures, and send high-quality context back to AI agents.

## Goals

- Provide a complete AI coding workbench that can be used as the main editor.
- Favor mature, maintained plugins over local reinvention.
- Keep local code focused on workflow glue that existing plugins do not provide.
- Make AI context transfer a first-class feature: git diffs, changed files, diagnostics, quickfix, DAP state, tests, tasks, and outline should be easy to send to AI.
- Keep startup and editing responsive, including for large files.
- Keep the configuration readable and debuggable by separating core modules from plugin specs.

## Non-Goals

- Do not build a general Neovim distribution.
- Do not add dashboard or ceremony-only UI.
- Do not add `which-key.nvim`.
- Do not add snippet bundles for the initial design; snippets are lower priority in this AI-first workflow.
- Do not use `snacks.nvim` as a general utility layer.
- Do not add database, Docker, Kubernetes, remote, or devcontainer UI in this project.
- Do not bind Sidekick to tmux in this Neovim design. Tmux remains an external workflow layer for later.

## Architecture

The configuration is organized as core modules plus plugin specifications.

```text
init.lua
lua/ad/
  bootstrap.lua
  options.lua
  autocmds.lua
  keymaps.lua

  core/
    bigfile.lua
    buffers.lua
    health.lua
    root.lua
    ui.lua

  ai/
    sidekick.lua
    contexts/
      git.lua
      review.lua
      debug.lua
      test.lua
      outline.lua

  plugins/
    init.lua
    ai.lua
    buffer.lua
    cmp.lua
    dap.lua
    editor.lua
    git.lua
    lsp.lua
    markdown.lua
    picker.lua
    test.lua
    treesitter.lua
    ui.lua
```

`core/*` owns editor-wide policy, such as bigfile detection, root detection, buffer helpers, health checks, and shared UI helpers.

`plugins/*` owns lazy.nvim specs and plugin-local configuration.

`ai/contexts/*` extends Sidekick context with workflow-specific context. It does not replace Sidekick's native context system.

`keymaps.lua` owns global keymap conventions. Plugin-specific keymaps should live near the plugin spec when that is clearer.

## AI Workflow

AI usage is split into four roles.

1. Small completion: `copilot.lua` integrates with `blink.cmp` for local code suggestions.
2. Context Q&A: `sidekick.nvim` sends selections, functions, files, diagnostics, quickfix, and buffers to external AI agents.
3. Review after AI edits: custom Sidekick contexts gather changed files, diffs, diagnostics, quickfix, and outline into `review_pack`.
4. Debug and test feedback: custom contexts gather DAP state, recent test output, recent task output, and diagnostics into `debug_pack`.

Neovim is the state organizer. External agents such as Codex and OpenCode remain responsible for large code-generation and refactoring tasks.

## Custom AI Contexts

`git_changed` returns a concise changed-file list from git.

`git_diff` returns the current repository diff or current-file diff, with size guards to avoid sending huge content accidentally.

`outline` returns an Aerial/LSP symbol summary for the current buffer.

`review_pack` combines changed files, diff, diagnostics, quickfix, and outline. It is intended for prompts such as: "Review the current AI-generated changes for bugs, syntax errors, unnecessary edits, missing tests, and risky behavior."

`dap_state` returns current debug session state when available: breakpoints, current stack frame, selected scopes or variables, and recent DAP output. If no session is active, it returns a clear no-data message.

`test_output` returns the most recent neotest failure summary when available.

`task_output` returns recent Overseer task output when available.

`debug_pack` combines diagnostics, quickfix, DAP state, test output, and task output for failure analysis.

All custom contexts must fail soft. Missing git state, inactive DAP sessions, or absent test output must not throw errors.

## Bigfile Strategy

Large-file handling is a local core module, not an external plugin.

`core.bigfile` follows the mature distribution pattern used by AstroNvim and LazyVim: detect large files centrally, mark the buffer, fire an event, and let expensive features downgrade per buffer.

Detection thresholds:

- File size greater than `1.5MB`
- Line count greater than `100000`
- Average line length greater than `1000`

When triggered:

- Set `vim.b.bigfile = true`
- Emit `User BigFile`
- Show `BIG` in the statusline
- Disable or downgrade Treesitter, semantic tokens, diagnostics, completion, format-on-save, fold, statuscolumn, listchars, render-markdown, Aerial, and DAP virtual text.

Bigfile mode should preserve basic editing, saving, grep, fzf search, git diff, and read-only inspection. Manual override commands can be added during implementation if needed.

## Review Workflow

After AI modifies code:

1. Use `barbar.nvim` to see open modified buffers and diagnostics.
2. Use `gitsigns.nvim` for line-level changes.
3. Use `codediff.nvim` for detailed diff review.
4. Use `neogit` for repository status and commit-oriented actions.
5. Use `trouble.nvim` and `quicker.nvim` for diagnostics and quickfix.
6. Use `review_pack` to send the current change set to an external AI agent for a second review.

## Debug and Test Workflow

For failing code:

1. Run tests through `neotest`.
2. Run project tasks through `overseer.nvim`.
3. Debug with `nvim-dap`, `nvim-dap-ui`, and `nvim-dap-virtual-text`.
4. Send `debug_pack` to AI when the failure needs assistance.

DAP and test integrations must not make startup fragile. Missing adapters should be reported through health checks instead of breaking editor startup.

## Health Checks

`core.health` should provide a local command or health provider that checks:

- Neovim version
- `git`
- `rg`
- `fd`
- `fzf`
- `node`
- `codex`
- Copilot authentication and LSP state where accessible
- Mason packages and DAP adapters
- Sidekick provider availability

Missing optional tools should be warnings. Missing core tools should be errors or strong warnings with clear remediation text.

## Validation Plan

Implementation should pass these gates:

1. Plugin installation: `nvim --headless "+Lazy! sync" +qa`
2. Startup health: `nvim --headless "+checkhealth" +qa`
3. Basic editing: file open, buffer switch, Oil, fzf-lua, barbar, Flash, theme, lualine, icons, and fidget.
4. Language features: LSP attaches, `blink.cmp` works, Copilot source appears, format and lint do not throw configuration errors.
5. Review features: gitsigns, neogit, codediff, diagnostics, quickfix, and `review_pack` work on a repository with changes.
6. Debug/test/task: DAP UI opens, neotest loads without adapter crashes, Overseer can run a simple task, and `debug_pack` fails soft when no session exists.
7. Markdown: render-markdown works inside Neovim and live-preview can open a browser preview.
8. Bigfile: opening a file above the configured threshold marks the buffer as bigfile and disables expensive features.

## Design Rules for Implementation

- Check current plugin docs and source before writing detailed configuration and keymaps.
- Prefer mature plugin APIs over local reimplementation.
- Keep local modules small and explicit.
- Avoid hard failure when optional external tools are missing.
- Keep keymaps consistent, but defer final keymap choices to implementation after checking plugin command names and APIs.
- Do not touch unrelated untracked files such as `.superpowers/`.
